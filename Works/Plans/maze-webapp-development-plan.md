# 2D Maze Web App Development Plan

## 1. 目的とスコープ
本計画書は、ブラウザ上で動作する 2D 迷路の自動生成および経路探索可視化 Web アプリケーションを、クライアントサイドのみで実装するための設計方針と作業分解を定義するものである。

今回のスコープは実装前計画に限定する。HTML、CSS、JavaScript の実装コード、README、LICENSE、仕様書本体の作成は次フェーズで行う。

開発対象の主要目的は以下の 3 点である。
- 完全迷路を安定して生成できること
- 探索過程と最短経路を視覚的に理解しやすく表示できること
- PC、タブレット、スマートフォンで無理なく操作できること

## 2. 要件整理
### 2.1 機能要件
- Difficulty をプルダウンで選択できること
- Difficulty は奇数グリッドサイズを前提とすること
- Super Hard を含む 4 段階の Difficulty を提供すること
- Generate Maze 操作で迷路を再生成し、表示状態をリセットすること
- Start Exploration 操作で探索アニメーションを開始すること
- ゴール到達後、最短経路を別アニメーションで強調表示すること
- Difficulty などの設定を localStorage に保存し、再訪時に復元すること

### 2.2 非機能要件
- クライアントサイドのみで完結すること
- 外部ライブラリを使用しないこと
- Vanilla JavaScript で実装すること
- モバイルファーストのレスポンシブ UI とすること
- 迷路描画は画面サイズに応じて動的スケーリングすること
- ロジックと描画を分離し、保守性と拡張性を確保すること

### 2.3 想定ユースケース
1. ユーザーが Difficulty を選択する
2. 初期表示時に localStorage から Difficulty を復元し、その値で迷路を自動生成する
3. ユーザーが Start Exploration を押して探索を開始する
4. 探索済みセルが順次表示される
5. ゴール到達後、最短経路が強調表示される
6. ページ再読み込み後も前回の Difficulty が復元される

## 3. 想定 UI 構成
### 3.1 画面構成
- ヘッダー: アプリタイトル
- コントロール領域: Difficulty セレクト、Generate Maze ボタン、Start Exploration ボタン
- ステータス領域: 現在の状態、グリッドサイズ、探索進行の要約を横長でコンパクト表示
- ビジュアライズ領域: 画面内で最優先に広く確保する迷路描画エリア

### 3.2 UI 文言
UI は英語表記とする。想定文言は以下。
- Title: Maze Forge: Generator & Pathfinding Explorer
- Difficulty
- Generate Maze
- Start Exploration
- Super Hard
- Ready
- Generating...
- Exploring...
- Path Highlighted

### 3.3 操作ルール
- 探索中は Difficulty 変更と Generate Maze を一時無効化する
- 初期表示では迷路の自動生成完了後に Start Exploration を有効化する
- 最短経路ハイライト中は全操作を一時無効化する
- 再生成時は探索結果、最短経路表示、内部状態を初期化する
- Status は迷路の右側ではなく上部に配置し、迷路の可視領域を圧迫しないことを優先する

## 4. 採用アルゴリズム候補と採用理由
### 4.1 迷路生成
採用候補は Recursive Backtracker を第一候補とする。

選定理由は以下。
- 完全迷路を確実に生成できる
- 実装が比較的単純で、Vanilla JavaScript と相性が良い
- 一本道と分岐のバランスが良く、視覚的に分かりやすい迷路になりやすい
- セル単位で生成過程を将来的に可視化しやすい

比較上の補足として、Prim 法や Kruskal 法も完全迷路を生成可能だが、今回の要件では見通しの良い実装と保守性を優先し、Recursive Backtracker が最適である。

### 4.2 経路探索
採用候補は A* アルゴリズムを第一候補とする。

選定理由は以下。
- 最短経路を保証できる
- ゴール方向への誘導が効くため、探索量を抑えやすい
- 探索過程の可視化に向いている
- グリッド迷路との親和性が高い

ヒューリスティックには Manhattan distance を採用する。上下左右移動のみのグリッドに適しており、実装と説明が明快である。

## 5. 描画方式の比較と採用方針
### 5.1 canvas 採用案
利点:
- セル数が増えても DOM 要素数が増えず、描画性能が安定しやすい
- 探索アニメーションと最短経路ハイライトを高速に描画しやすい
- リサイズ時の再描画制御がしやすい

注意点:
- 個々のセルを DOM として直接扱えないため、描画状態を内部データで厳密に管理する必要がある

### 5.2 DOM 描画案
利点:
- 構造が直感的で、初期実装が理解しやすい
- 開発時の見た目確認がしやすい

注意点:
- 101x101 以上ではセル数増加に伴う更新コストが高くなりやすい
- アニメーション中の頻繁なクラス更新でパフォーマンスが不安定になりやすい

### 5.3 採用方針
描画方式は canvas を採用する。

理由は以下。
- 難易度 Hard でも描画パフォーマンスを維持しやすい
- Super Hard でも DOM 描画より性能と表示密度を確保しやすい
- 迷路全体の再描画と部分更新の制御が容易
- レスポンシブ対応時にセルサイズ再計算と再描画で整合を取りやすい

## 6. クラス設計方針と責務分担
### 6.1 MazeGenerator
責務:
- 指定サイズの空グリッドを作成する
- Recursive Backtracker で完全迷路を生成する
- スタート位置とゴール位置を決定する
- 通路と壁の構造を返す

主な入出力:
- 入力: グリッドサイズ
- 出力: 迷路データ、開始位置、終了位置

### 6.2 MazeSolver
責務:
- A* による探索を行う
- 探索済み順序を記録する
- 親参照を保持し、ゴール後に最短経路を復元する

主な入出力:
- 入力: 迷路データ、開始位置、終了位置
- 出力: 訪問順序、最短経路、探索統計

### 6.3 MazeRenderer
責務:
- canvas の初期化とリサイズ対応を行う
- 迷路全体を描画する
- 探索中セル、開始点、終了点、最短経路を描画する
- 状態ごとに色分けルールを適用する

主な入出力:
- 入力: 迷路データ、描画状態、表示サイズ
- 出力: 画面描画

### 6.4 AppController
責務:
- UI イベントを管理する
- localStorage の保存と復元を行う
- 生成、探索、描画の処理順を制御する
- 実行状態に応じて UI の有効無効を切り替える

### 6.5 補助クラスまたは補助モジュール候補
- StorageService: localStorage への入出力を分離する場合に追加
- AnimationController: 探索表示と最短経路表示の待機制御を分離する場合に追加
- Config: Difficulty 定義や色設定を集約する場合に追加

初期実装ではファイル数を抑えるため AppController 内の private 相当メソッドで開始し、肥大化した時点で補助クラスへ分離する方針が現実的である。

## 7. データ構造の方針
### 7.1 迷路表現
内部表現は 2 次元配列を採用する。

セル値の候補:
- 0: wall
- 1: path

別管理する状態:
- start position
- goal position
- visited order
- final path
- rendered visited count
- rendered path count
- current status

探索アニメーションとリサイズ再描画の整合を取るため、論理状態と描画進行状態を分離する。visited order と final path は完全な計算結果を保持し、rendered visited count と rendered path count はその時点までに画面へ反映済みの範囲を表す。

### 7.2 座標表現
座標は row, col のオブジェクトで統一する。

理由:
- 可読性が高い
- 描画座標との変換が明確
- 探索ロジックのバグを追いやすい

### 7.3 探索用データ
- open set: 優先度付き配列で管理
- closed set: Set で管理
- cameFrom: 親参照マップ
- gScore, fScore: Map で管理

セル数が限定的なため、初期実装ではシンプルな配列ベースの open set でも許容可能である。ただし保守性を考慮し、最小 fScore の取得を関数化しておく。

## 8. localStorage で保存する状態の一覧
保存対象は以下とする。
- selectedDifficulty

保存しない対象は以下とする。
- 現在の迷路構造
- 探索途中状態
- 最短経路表示状態
- lastGridSize

理由:
- グリッドサイズは selectedDifficulty から一意に導出できる
- 保存値を 1 系統に絞ることで復元時の不整合を防げる
- 迷路そのものを復元対象にすると状態整合が複雑になる
- 再訪時は設定のみ復元し、迷路は新規生成した方が UI と内部状態が安定する

キー名候補:
- mazeApp.selectedDifficulty

## 9. レスポンシブ対応方針
### 9.1 レイアウト方針
- モバイルファーストで縦積みレイアウトを基本とする
- 画面幅拡大時にコントロール領域を横並びへ拡張する
- ステータス領域は迷路の上に配置し、横長のコンパクトな情報帯として扱う
- 迷路表示領域は常に正方形を維持する
- 迷路表示領域はページ内で最も大きい視覚要素として確保する

### 9.2 スケーリング方針
- canvas の描画領域は親コンテナ幅を最大限使用して決定する
- 使用可能な viewport 高さも考慮し、はみ出さない最大正方形サイズを計算する
- セルサイズは canvasSize / gridSize で求める
- リサイズ発生時は迷路を再生成せず、保持済みの論理状態と描画進行状態から再描画する

### 9.3 モバイル操作配慮
- ボタンのタップ領域を十分確保する
- セレクト要素の高さと余白を広めに取る
- テキストと描画色のコントラストを確保する
- Status は 1 行または 2 行で収まる情報量に絞り、縦方向の占有を抑える

## 10. アニメーション実装方針
### 10.1 探索アニメーション
- MazeSolver が返した訪問順序を使用する
- AppController が async/await で順次描画を進める
- 一定ステップごとに await でブラウザへ制御を返す
- 描画ごとに rendered visited count を更新し、途中リサイズ時も同じ進行度を再現できるようにする

### 10.2 最短経路ハイライト
- 探索完了後に path 配列を順次描画する
- 探索中とは別色、別テンポで表示し、状態変化を明確にする
- 描画ごとに rendered path count を更新し、途中リサイズ時も同じ進行度を再現できるようにする

### 10.3 ブロッキング回避
- 1 セルごとの完全待機ではなく、複数セル単位の描画バッチを検討する
- 難易度 Hard では描画間隔を自動調整できる設計が望ましい
- リサイズイベント発生時は現在の描画進行度を用いて再描画し、探索計算自体は再実行しない

初期実装では分かりやすさを優先し、固定ディレイで開始する。必要に応じて後続フェーズで速度制御を追加する。

## 12. テスト観点
### 12.1 機能テスト
- Difficulty 変更でグリッドサイズが切り替わるか
- Super Hard を選択しても迷路生成と探索が成立するか
- Generate Maze で毎回迷路が再生成されるか
- Start Exploration で探索が始まるか
- ゴール後に最短経路が表示されるか

### 12.2 アルゴリズム検証
- 生成迷路が常にスタートからゴールまで到達可能か
- A* の結果が最短経路になっているか
- 壁セルを経路として通過していないか

### 12.3 状態管理テスト
- localStorage に Difficulty が保存されるか
- リロード後に UI が復元されるか
- 生成時に探索状態が正しく初期化されるか
- 探索途中または最短経路ハイライト途中でリサイズしても描画進行度が破綻しないか

### 12.4 UI テスト
- スマートフォン幅でレイアウトが破綻しないか
- タブレットと PC 幅で描画サイズが適切か
- Status が上部でコンパクトに表示され、迷路領域を圧迫していないか
- 迷路表示領域が従来より十分大きく確保されているか
- ボタンが押しやすく、表示が判読可能か

### 12.5 パフォーマンステスト
- Hard サイズで操作不能な遅延が出ないか
- Super Hard サイズで致命的な描画遅延が出ないか
- 探索アニメーション中に表示更新が極端に詰まらないか
- リサイズ時に不要な再生成が発生しないか

## 13. リスクと注意点
- Hard サイズで 1 セルごとの遅延が長すぎると体感が悪化する
- Super Hard ではセルサイズが小さくなり視認性が落ちやすい
- canvas リサイズ時に内部座標と表示サイズの不整合が起きやすい
- A* の open set 実装を単純化しすぎると探索性能が落ちる
- 探索中に再操作を許可すると状態競合が起きやすい
- 描画進行状態を持たないとリサイズ時にアニメーション途中の見た目を復元できない

対策方針:
- 実行状態フラグを AppController で一元管理する
- 迷路データと描画状態を分離する
- リサイズ時は再描画のみ行い、迷路再生成や探索再計算は行わない
- 保存対象は selectedDifficulty に限定する
- 描画進行度をカウント値として保持する
- Status を圧縮し、迷路表示領域を優先してレイアウトする

## 14. 実装開始前の確認事項
- スタート位置とゴール位置を固定にするか、外周上で可変にするか
- 色設計をシンプル重視にするか、教育用可視化として強いコントラストを持たせるか
- 将来的に BFS や DFS など複数探索アルゴリズム比較へ拡張する余地を残すか

本変更では以下を確定事項とする。
- タイトルは Maze Forge: Generator & Pathfinding Explorer とする
- Difficulty は Easy = 25x25、Normal = 51x51、Hard = 101x101、Super Hard = 201x201 とする
- Status は迷路の上部に配置する
- Status はコンパクト表示を優先し、迷路表示領域を最大化する

## 16. 実装着手順の推奨順序
実装時は以下の順序を推奨する。

1. HTML の UI 骨格を作る
2. canvas をレスポンシブに表示する
3. MazeGenerator で静的迷路を描画する
4. MazeSolver で経路計算結果を取得する
5. 探索アニメーションを追加する
6. 最短経路ハイライトを追加する
7. localStorage 復元と UI 制御を仕上げる
8. 最後に README、LICENSE、仕様書を整備する

この順序により、描画、生成、探索、UI 制御を段階的に検証できるため、原因切り分けが容易になる。
