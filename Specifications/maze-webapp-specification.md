# 2D Maze Web App Specification

## 1. 文書概要
### 1.1 文書の目的
本仕様書は、ブラウザ上で動作する 2D 迷路の自動生成および経路探索可視化 Web アプリケーションの正式仕様を定義するものである。実装担当者が本書のみを参照して開発を進められることを目的とする。

### 1.2 対象システム
対象システムは、HTML5、CSS3、JavaScript ES6+ を用いて実装されるクライアントサイド完結型の Web アプリケーションである。

### 1.3 スコープ
本仕様書の対象範囲は以下とする。
- 画面仕様
- 機能仕様
- 描画仕様
- 状態管理仕様
- データ仕様
- 非機能要件
- 受け入れ基準

本仕様書の対象外は以下とする。
- サーバーサイド処理
- 外部ライブラリ導入
- ユーザー認証
- バックエンド API 連携

## 2. システム概要
### 2.1 アプリケーション概要
本アプリケーションは、完全迷路を自動生成し、スタートからゴールまでの経路探索過程および最短経路をブラウザ上で可視化する。

### 2.2 主な利用目的
- 迷路生成アルゴリズムの視覚的理解
- 経路探索アルゴリズムの視覚的理解
- PC、タブレット、スマートフォンにおける軽量な教育用・デモ用利用

### 2.3 前提条件
- JavaScript が有効なブラウザで実行されること
- localStorage が利用できる場合は設定保存機能を利用すること
- ネットワーク接続は必須ではないこと

## 4. 画面仕様
### 4.1 画面構成
画面は以下の領域で構成する。
- ヘッダー領域
- コントロール領域
- ステータス領域
- 迷路表示領域

### 4.2 ヘッダー領域
- アプリタイトルを表示すること
- 想定文言は Maze Forge: Generator & Pathfinding Explorer とすること

### 4.3 コントロール領域
以下の UI 要素を配置すること。
- Difficulty 選択プルダウン
- Generate Maze ボタン
- Start Exploration ボタン

### 4.4 迷路表示領域
- 迷路は正方形領域内に描画すること
- 描画方式は canvas を採用すること
- 表示領域は画面サイズに応じて動的に変化すること
- 画面内で最優先に大きく確保すること

### 4.5 ステータス領域
以下の情報を表示可能とすること。
- 現在の状態
- 選択中 Difficulty
- グリッドサイズ
- 探索完了状態
- ステータス領域は迷路の右側ではなく上部に配置すること
- ステータス領域は横長でコンパクトな表示とし、迷路表示領域を圧迫しないこと

### 4.6 UI 表示文言
UI 表示文言は英語とする。最低限の文言は以下とする。
- Maze Forge: Generator & Pathfinding Explorer
- Difficulty
- Generate Maze
- Start Exploration
- Ready
- Generating...
- Exploring...
- Path Highlighted
- Super Hard

## 5. 機能仕様
### 5.1 初期表示
- アプリ起動時に Difficulty 設定を localStorage から読み込むこと
- localStorage に値が存在しない場合はデフォルト Difficulty を適用すること
- 初期表示時に迷路を自動生成すること
- 初期表示完了後、Start Exploration が実行可能であること

初期表示時は、復元した Difficulty を正としてグリッドサイズを導出すること。

### 5.2 Difficulty 設定機能
#### 5.2.1 概要
ユーザーはプルダウンから迷路サイズを選択できること。

#### 5.2.2 選択肢
- Easy = 25 x 25
- Normal = 51 x 51
- Hard = 101 x 101
- Super Hard = 201 x 201

#### 5.2.3 制約
- グリッドサイズは奇数のみを許可すること
- 迷路生成中は変更不可とすること
- 探索中は変更不可とすること
- 最短経路ハイライト中は変更不可とすること

#### 5.2.4 動作
- Difficulty 変更時に selectedDifficulty を localStorage に保存すること
- Difficulty 変更後は新しいサイズで迷路を生成し直すこと
- 変更時には探索状態と最短経路表示状態を初期化すること

### 5.3 Generate Maze 機能
#### 5.3.1 概要
ユーザー操作により迷路を新規生成する機能を提供すること。

#### 5.3.2 動作
- ボタン押下時に現在の Difficulty に対応するサイズで迷路を再生成すること
- 再生成時に探索結果を破棄すること
- 再生成時に最短経路表示を破棄すること
- 再生成後は Ready 状態に戻ること
- 迷路生成中は Generate Maze を再度実行不可とすること
- 生成処理が失敗した場合は、直前まで表示していた迷路と描画済み進行状態を維持すること
- 生成処理が失敗した場合は、部分的に生成された迷路や不整合な中間状態へ置き換えてはならないこと

### 5.4 Start Exploration 機能
#### 5.4.1 概要
生成済み迷路に対して経路探索を実行し、その過程をアニメーション表示すること。

#### 5.4.2 動作
- ボタン押下時に A* による探索を開始すること
- 探索済みセルを時間経過に沿って順次描画すること
- 探索中は Difficulty プルダウンと Generate Maze ボタンを無効化すること
- 探索完了後に最短経路ハイライトへ遷移すること
- 初期表示で自動生成が完了していない場合は実行不可とすること

### 5.5 最短経路ハイライト機能
#### 5.5.1 概要
ゴール到達後に、スタートからゴールまでの最短経路を別アニメーションで表示すること。

#### 5.5.2 動作
- 探索完了後にのみ実行すること
- 最短経路セルを探索済みセルと異なる色で表示すること
- 最短経路表示完了後、状態を Path Highlighted とすること
- ハイライト中は全操作を無効化すること

## 6. 状態遷移仕様
### 6.1 状態一覧
アプリケーション状態は以下を持つこと。
- idle
- generating
- ready
- exploring
- highlighting
- completed

### 6.2 状態の意味
- idle: 初期化前
- generating: 迷路生成中
- ready: 迷路生成済みで探索開始可能
- exploring: 探索アニメーション中
- highlighting: 最短経路ハイライト中
- completed: 最短経路表示完了後

### 6.3 状態遷移
- idle -> generating
- generating -> ready
- ready -> exploring
- exploring -> highlighting
- highlighting -> completed
- completed -> generating
- ready -> generating

### 6.4 状態別 UI 制御
- generating 中は Difficulty、Generate Maze、Start Exploration を無効化すること
- exploring 中は Difficulty と Generate Maze を無効化すること
- highlighting 中は全操作を無効化すること
- ready と completed では操作可能状態へ戻すこと

## 7. アルゴリズム仕様
### 7.1 迷路生成アルゴリズム
迷路生成には Recursive Backtracker を採用すること。

#### 採用理由
- 完全迷路を生成できること
- 実装が比較的単純であること
- 生成結果が視覚的に分かりやすいこと

#### 要件
- 生成される迷路は必ず解を持つこと
- 壁と通路をグリッドサイズ x グリッドサイズ要素数の 1 次元配列で表現すること
- スタートからゴールまで到達可能であること

### 7.2 経路探索アルゴリズム
経路探索には A* アルゴリズムを採用すること。

#### 採用理由
- 最短経路を保証できること
- グリッド迷路との相性が良いこと
- 可視化時に探索の進み方が理解しやすいこと

#### ヒューリスティック
- Manhattan distance を使用すること
- 移動方向は上下左右のみとすること

#### 要件
- 壁セルを通過しないこと
- ゴール到達後に親参照を用いて最短経路を復元すること
- 探索済み順序を可視化用に保持すること

## 8. データ仕様
### 8.1 迷路データ
迷路はグリッドサイズ x グリッドサイズ要素数を持つ 1 次元配列で表現すること。

セル値定義:
- 0 = wall
- 1 = path

配列インデックスは cellId として扱い、row = floor(cellId / gridSize)、col = cellId % gridSize により論理座標へ変換可能であること。

### 8.2 座標データ
座標は cellId を主表現とし、必要に応じて row と col を導出して扱うこと。

- cellId
- row
- col

### 8.3 内部状態データ
最低限、以下の状態を保持すること。
- mazeGrid
- startId
- goalId
- visitedOrder
- shortestPath
- renderedVisitedCount
- renderedPathCount
- selectedDifficulty
- currentStatus

mazeGrid は 1 次元配列、startId と goalId はセル ID とすること。

visitedOrder と shortestPath はセル ID 列として計算結果全体を保持し、renderedVisitedCount と renderedPathCount は画面へ反映済みの進行度を表すこと。

### 8.4 探索補助データ
探索時は以下を保持すること。
- openSet
- closedSet
- cameFrom
- gScore
- fScore

探索補助データは cellId をキーまたは値として扱える構造で保持すること。

## 9. 永続化仕様
### 9.1 保存対象
localStorage に保存する対象は以下とする。
- mazeApp.selectedDifficulty

### 9.2 保存タイミング
- Difficulty 変更時

### 9.3 復元タイミング
- 初回ロード時
- ページリロード時

### 9.4 保存対象外
以下は localStorage に保存しないこと。
- 迷路構造そのもの
- 探索途中の状態
- 最短経路表示結果
- lastGridSize

グリッドサイズは selectedDifficulty から導出すること。復元時に保存済みグリッドサイズを参照してはならない。

## 10. 描画仕様
### 10.1 描画方式
- canvas を使用すること
- 迷路全体の再描画が可能であること

### 10.2 色分け
最低限、以下の視覚区別が必要である。
- 壁
- 通路
- スタート
- ゴール
- 探索済みセル
- 最短経路

### 10.3 リサイズ対応
- 画面リサイズ時に canvas サイズを再計算すること
- リサイズ時に迷路再生成は行わず、再描画のみ行うこと
- リサイズ時に探索計算や最短経路計算を再実行しないこと
- 保持済みの論理状態と描画進行度から同じ表示状態を再構築すること
- 迷路表示領域は常に正方形を維持すること
- 迷路表示領域は利用可能な幅と高さのうち最大限を使用すること

### 10.4 スケーリング
- 表示可能領域内で最大サイズの正方形を使用すること
- セルサイズは canvas サイズとグリッドサイズから算出すること
- Super Hard でも迷路全体が表示領域内に収まること

## 11. レスポンシブ仕様
### 11.1 レイアウト方針
- モバイルファーストで実装すること
- 狭い画面ではヘッダー、コントロール、ステータス、迷路の順で縦積みレイアウトとすること
- 広い画面でもステータスは迷路の上部に配置すること
- 広い画面ではコントロールを横並びにできること

### 11.2 モバイル操作要件
- ボタンはタップしやすいサイズであること
- プルダウンは指操作しやすい高さを確保すること
- スクロールなしでも主要操作にアクセスしやすい構成とすること
- ステータスは短いラベルと短い値で構成し、縦方向の占有を最小化すること

### 11.3 視認性要件
- テキストと背景のコントラストを確保すること
- 探索済みセルと最短経路セルが明確に識別できること
- 迷路表示領域は UI 内で最も視認性が高い主要領域であること

## 12. アニメーション仕様
### 12.1 実装方針
- async/await を用いて段階描画を行うこと
- ブラウザのメインスレッドを長時間占有しないこと
- 論理状態と描画進行状態を分離して管理すること

### 12.2 探索アニメーション
- visitedOrder を使用して順次描画すること
- 描画更新の間に制御を返すこと
- 描画ごとに renderedVisitedCount を更新すること
- リサイズ時は renderedVisitedCount 時点までの表示を再構築すること

### 12.3 最短経路アニメーション
- shortestPath を使用して順次描画すること
- 探索アニメーションとは異なる表示テンポまたは色を使用すること
- 描画ごとに renderedPathCount を更新すること
- リサイズ時は renderedPathCount 時点までの表示を再構築すること

### 12.4 パフォーマンス配慮
- Hard サイズでも極端なカクつきが発生しないこと
- Super Hard サイズでも操作不能な状態に陥らないこと
- 必要に応じて複数セル単位のバッチ描画を行える設計とすること
- 必要に応じて静的レイヤーキャッシュ、差分再描画、座標キャッシュ、Path2D を用いたバッチ描画を利用できる設計とすること
- 迷路生成および探索計算は、メインスレッド占有を避けるため Worker へ委譲できる設計とすること

## 13. アーキテクチャ仕様
### 13.1 設計方針
- ロジックと描画を分離すること
- モジュールまたはクラス単位で責務を明確化すること
- UI、状態管理、描画、Worker 通信、Worker 内アルゴリズムを分離すること

### 13.2 主要コンポーネント責務
#### AppController
- UI イベント、状態管理、永続化、描画要求、Worker 要求開始を担当すること

#### MazeRenderCanvas
- canvas 描画、再描画、リサイズ対応、静的レイヤーキャッシュを担当すること

#### MazeWorkerRequestClient
- Worker の初期化、要求送信、キャンセル、完了通知、失敗通知を担当すること

#### Worker Algorithms
- Recursive Backtracker による迷路生成を担当すること
- A* による経路探索と最短経路復元を担当すること

#### App State and UI Sync
- 状態オブジェクト生成、永続化補助、探索進行反映、UI の有効無効同期を担当すること

## 14. エラー処理仕様
以下のケースではアプリケーションが破綻しないこと。
- localStorage が利用できない場合
- 不正な Difficulty 値が保存されていた場合
- 探索開始前に迷路が未生成だった場合
- リサイズイベントが連続発生した場合
- Web Worker が利用できない場合
- 迷路生成処理または探索処理が失敗した場合

対処方針:
- 不正値はデフォルト Difficulty にフォールバックすること
- 永続化失敗時もアプリ本体は継続利用可能とすること
- localStorage が利用できない場合は保存機能のみ無効化し、実行時設定で継続利用可能とすること
- 実行不能時は状態を維持し、UI を壊さないこと
- リサイズ時は進行中アニメーションの計算結果を破棄しないこと
- Web Worker が利用できない場合は、ページ表示と UI は維持し、Worker を必要とする迷路生成および探索要求は失敗として扱うこと
- 迷路生成失敗時は、直前まで正常表示されていた旧迷路、探索結果、最短経路表示進行度を維持すること
- 失敗時に新しい Difficulty 表示やグリッドサイズ表示だけが先行して更新される不整合を生じさせてはならないこと

## 15. テスト観点
### 15.1 機能観点
- Difficulty 変更で迷路サイズが切り替わること
- Super Hard を選択できること
- Generate Maze で迷路が再生成されること
- Start Exploration で探索が開始されること
- 探索後に最短経路が表示されること
- 迷路生成失敗時に旧迷路表示が維持されること

### 15.2 アルゴリズム観点
- 生成迷路に必ず解が存在すること
- A* が最短経路を返すこと
- 壁を経由する経路が生成されないこと

### 15.3 UI 観点
- スマートフォン表示で操作不能にならないこと
- タブレット、PC 表示でレイアウトが崩れないこと
- 探索済みセルと最短経路が視覚的に判別できること
- Status が迷路の上部でコンパクトに表示されること
- 迷路表示領域がページ内で十分大きく確保されること

### 15.4 永続化観点
- Difficulty が localStorage に保存されること
- リロード後に Difficulty が復元されること
- 復元後のグリッドサイズが Difficulty から正しく導出されること
- localStorage が利用できない環境でもアプリ本体の操作が継続可能であること

### 15.5 パフォーマンス観点
- Hard サイズで致命的な遅延が発生しないこと
- Super Hard サイズで致命的な遅延が発生しないこと
- リサイズ時に不要な迷路再生成が発生しないこと
- 探索途中またはハイライト途中のリサイズで表示進行度が破綻しないこと

## 16. 受け入れ基準
以下をすべて満たした場合、本仕様を満たしたと判断する。
- クライアントサイドのみで動作すること
- Easy、Normal、Hard、Super Hard の 4 段階 Difficulty を選択できること
- 迷路生成後に探索アニメーションを実行できること
- ゴール到達後に最短経路が強調表示されること
- Difficulty 設定が再訪時に復元されること
- スマートフォン、タブレット、PC で表示と操作が成立すること
- Status が上部にコンパクト表示され、迷路表示領域が十分大きいこと
- 外部ライブラリ未使用であること
